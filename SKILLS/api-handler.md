# API Handlers — BuildBridge

This document defines the standards for all API endpoints and server-side logic for the BuildBridge platform. Every handler must be built to support the project's mission of providing a safe, fraud-resistant, and trust-first environment for the Nigerian informal economy.


## 1. Core Requirements
* **Input Validation**: All API handlers must strictly validate and sanitize input before processing to treat client data as untrustworthy.

* **Structured Responses**: All responses must use structured JSON to ensure compatibility with mobile and web clients.

* **Standardized Errors**: Handlers must return appropriate HTTP status codes (e.g., 400 for bad requests, 404 for missing resources).

* **Security & Auth**: Unauthorized requests must return a 401 status code.

* **Access Control**: Access to admin and moderation workflows must be strictly separated with strong security rules.


## 2. Nigeria-Specific Constraints
* **Phone-First Authentication**: APIs must prioritize phone-based OTP for user identification and account recovery.

* **Normalization**: All Nigerian phone numbers must be normalized to a consistent international format (e.g., +234...) before database entry or messaging operations.

* **Data Minimization**: API payloads should be optimized for low-bandwidth and high-latency mobile connections.

* **NDPR Compliance**: Every handler interacting with personal data must ensure explicit consent has been captured and that data is handled according to Nigerian Data Protection Regulations.


## 3. Trust & Financial Logic
* **Idempotency**: All pledge creation and disbursement handlers must implement idempotency keys to prevent duplicate transactions caused by unstable network connections.

* **Fee Transparency**: Handlers calculating pledges must return a clear breakdown of fees and the exact amount the tradesperson will receive before final confirmation.

* **Escrow Management**: Payment handlers must support transparent escrow logic, ensuring funds are held securely until proof-of-use or moderation milestones are met.

* **Fraud Prevention**: Implement automated triggers for suspicious activity, such as duplicate listings or high-frequency vouching from the same device.


## 4. Messaging & Notifications
* **Opt-in/Opt-out**: Messaging operations (SMS/WhatsApp) must respect user consent and unsubscribe keywords.

* **Dignified Communication**: Automated notification copy generated by the backend must follow project vocabulary—using terms like "Pledge" and "Backer"—and avoid robotic or patronizing tones.


## 5. Failure Handling & States
Handlers must not remain silent during errors. They should return actionable data to the UI to handle the following BuildBridge-specific cases:
* **Missing Proof-of-Use**: Trigger moderation alerts if updates are not submitted post-funding.
* **Disputed Vouchers**: Flag accounts for manual review when a community vouch is contested.
* **Under-Review States**: Return specific status codes when a listing or profile is pending moderation to keep the user informed.


## 6. Implementation Example
All handlers should be organized into modules such as `onboarding`, `listings`, `payments`, and `impact` to maintain a maintainable codebase.

## 7. Payments & Robustness

This document extends the core API Handlers standard to include specific requirements for payment processing, escrow management, and operational robustness within the BuildBridge ecosystem.

### 1. Payment & Pledging Rules

* **Atomic Transactions**: All pledge operations must be atomic. The creation of a "Pledge" record and the initiation of the payment gateway session must happen in a way that prevents "ghost pledges" (pledges that exist without payment attempts).

* **Provider Abstraction**: Use a factory pattern for payment providers (e.g., Paystack, Flutterwave). The core application logic should not be tightly coupled to a specific vendor's API structure.

* **Webhook Security**: 
    * All incoming webhooks from payment processors must be verified using secret hash signatures. 
    * Handlers must return a `200 OK` immediately to the provider before processing logic to prevent timeout retries.

* **Escrow Logic**: APIs must support a "Held" state. Funds should only transition from `Escrow` to `Disbursed` upon successful verification of a "Proof-of-Use" update or a manual moderation override.
* **Currency Safety**: All financial values must be handled as **integers (minor units/kobo)** to avoid floating-point errors. APIs must explicitly return the currency code (e.g., `NGN`) with every amount.


# API Handlers: Server-Side Payment & Security — BuildBridge

This document mandates that all financial logic and payment orchestration occur strictly on the server to prevent client-side tampering.


### Server-Side Payment Orchestration

* **No Client-Side Totals**: The frontend must never calculate the final payment amount. The server is the sole authority for calculating pledges, fees, and currency conversions.

* **Tokenized Initialization**: The frontend only requests a "Payment Session." The server communicates with the payment provider (e.g., Paystack/Flutterwave), generates a secure transaction URL or token, and passes only that back to the client.

* **Direct Webhook Processing**: Success or failure is determined strictly by server-to-server webhooks. The frontend "Success Page" is treated as a visual courtesy; the actual "Pledge Verified" state is only updated once the server validates the provider's signature.

### Transaction Atomicity & Integrity

* **Database Transactions**: All pledge-related writes must be wrapped in a database transaction block. If any part of the record-keeping fails (Pledge record, Ledger update, or Notification trigger), the entire operation must rollback.

* **The "Double-Spend" Guard**: Every payment request must include an `idempotency_key` generated by the server. This prevents a user with a shaky connection from accidentally being charged twice if they click "Pay" multiple times.

### Escrow & Disbursement Rules

* **Locked States**: Funds must be programmatically locked in an `ESCROW_HELD` state upon payment confirmation.

* **Proof-Validated Release**: The API for disbursement must require a valid `proof_of_use_id` and a `moderation_status === 'APPROVED'` before moving funds from Escrow to the tradesperson’s payout account.

* **Audit Trail**: Every movement of funds (Inbound -> Escrow -> Outbound) must generate an immutable log entry including the timestamp, transaction hash, and associated User IDs.

### Security Fail-Safes

* **Environment Isolation**: Payment secret keys must never exist in the frontend code or be exposed via any public API response.

* **Signature Verification**: Every incoming webhook must undergo a cryptographic signature check against the provider’s public key before the payload is trusted.
* **Rate Limiting**: Payment initialization endpoints must be strictly rate-limited to prevent automated "card testing" or brute-force attacks on the payment gateway.

### Missing Robustness Rules 

### A. Rate Limiting & Anti-Abuse
* **OTP Throttling**: Because we use phone-first authentication, API handlers for OTP requests must be strictly rate-limited by IP and Phone Number to prevent SMS toll fraud and exhaustion of messaging credits.
* **Vouching Limits**: To prevent "trust-farming," limit the number of vouches a single account can provide within a 24-hour window.

### B. Offline-First & Sync Logic
* **Optimistic UI Support**: APIs should return the "expected" state of a resource immediately, allowing the mobile client to show changes before the database confirms (especially critical for low-bandwidth areas).
* **Conflict Resolution**: Handlers for "Need" creation must handle cases where a user might have submitted the same form twice due to a "hanging" connection.

### C. Media Handling (Proof-of-Use)
* **Asynchronous Processing**: Handlers for Proof-of-Use uploads (images/video) must handle file metadata first and process the actual binary data via a background worker to prevent blocking the main API thread.
* **Compression Enforcement**: The API must reject raw images exceeding specific sizes, forcing the client or an intermediary worker to compress assets for low-bandwidth viewers.

### D. Audit Logging
* **Immutable Logs**: Every state change for a "Need" (e.g., `Draft` -> `Pending Review` -> `Live` -> `Funded`) must be logged with a timestamp and the ID of the performing agent (User or Admin). These logs must be read-only.


### Financial Vocabulary Enforcement
All payment-related API responses must use the semantic keys defined in the project Guide:
* `total_pledged` (not total_donated)
* `backer_count` (not donor_count)
* `disbursement_status` (not payout_status)
* `platform_fee` (must be explicitly calculated and returned for transparency)


## 4. Error Codes: Financial Context
| Code | Meaning | User-Facing Action |
| :--- | :--- | :--- |
| `ERR_PAYMENT_IDEMPOTENCY` | Duplicate transaction detected | "This pledge was already processed." |
| `ERR_INSUFFICIENT_VOUCHES` | Need cannot go live yet | "You need 2 more vouches from your community." |
| `ERR_PROOF_REQUIRED` | Cannot disburse funds | "Please upload your Proof-of-Use to unlock funds." |
| `ERR_PROVIDER_DOWN` | Payment gateway issue | "Payment service is currently slow. Please try again." |
